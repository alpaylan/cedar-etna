# cedar-etna — ETNA workload for Cedar-Lean

ETNA workload built from the [`cedar-policy/cedar-spec`](https://github.com/cedar-policy/cedar-spec) Lean 4 formalization. Each variant reintroduces one historical bug fix from cedar-spec history by reverse-applying a hand-crafted patch against a pinned base commit, paired with a property and a deterministic witness. Patches are the only durable per-variant artefact — no `etna/<variant>` git branches.

The workload exercises Cedar's typechecker, validator, request-validator, level checker, and SymCC encoder. 6 of the 9 variants are *type-system / soundness* bugs; 2 are parser bugs; 1 is a level-checker completeness bug.

## Prerequisites

- [`elan`](https://github.com/leanprover/elan) — Lean toolchain manager. The workload pins `leanprover/lean4:v4.29.1` via `lean-toolchain`.
- [`etna`](https://github.com/alpaylan/etna) (the etna2 CLI) — for `etna workload doc` regeneration. Optional; the workload still runs without it via the validate script directly.
- ~3 GB of disk for the `.cedar-spec` checkout + Lean build artefacts.
- macOS / Linux. The validate script uses `bash`, `git apply`, `lake`, and Python 3.11+ (for `tomllib`).

## Layout

```
workloads/Lean/Cedar/
├── .cedar-spec/                 # sparse checkout of cedar-policy/cedar-spec, etna-base branch
│   └── cedar-lean/
│       ├── Cedar/Etna/          # property definitions + witnesses (committed on etna-base)
│       │   ├── Property.lean    # PropertyResult inductive + emit_json
│       │   ├── Properties.lean  # property_<snake_case> functions, one per variant
│       │   └── Witnesses.lean   # witness_<name>_case_<tag> deterministic fixtures
│       ├── EtnaCedar/Main.lean  # `lake exe etna_cedar <tool> <Property>` dispatcher
│       └── lakefile.lean        # adds Plausible v4.29.0 dep + lean_exe etna_cedar
├── patches/                     # one .patch per variant, buggy → fixed direction
├── scripts/validate_lean_workload.sh
├── etna.toml                    # source of truth: variants, properties, witnesses, source PRs
├── BUGS.md, TASKS.md            # generated by `etna workload doc`
├── progress.jsonl               # per-stage event log (discover, atomize, runner, validate)
└── README.md                    # this file
```

The Cedar source is checked out via `git clone --no-checkout` + `git sparse-checkout` so the working tree is small. The `.cedar-spec` HEAD lives on the `etna-base` branch, which holds the runner scaffolding on top of upstream `main`. Variant patches apply against that branch's tip.

## Quickstart for sharing

Marauders' patch handling makes the full `etna experiment run` flow work:
patches at `patches/*.patch` are auto-discovered, `marauders set --variant
<name>` applies one, and etna-cli's driver calls `marauders::set_variant`
internally. The only piece etna-cli doesn't provide automatically is the
`lean` language declaration — the friend needs to drop a `marauder.toml`
at the experiment root (we ship one in this repo to copy).

### Recommended: full `etna experiment run`

```sh
# 1. Fresh experiment
etna experiment new my-cedar-experiment
cd my-cedar-experiment

# 2. Pull in cedar-etna + its auto-generated test descriptor
etna workload add https://github.com/alpaylan/cedar-etna.git

# 3. Tell etna-cli's marauders integration that 'lean' is a known language.
#    The cloned workload ships the right marauder.toml; copy it up one level.
cp workloads/cedar-lean/marauder.toml ./marauder.toml

# 4. Pick a strategy and run. 'plausible' is random search; 'etna' is the
#    deterministic witness replay.
etna experiment amend-test --test cedar-lean --strategy plausible
etna experiment run --tests cedar-lean

# 5. Look at results. store.jsonl has one row per (variant, trial); etna-cli's
#    visualize / report subcommands work directly on it.
etna experiment report --name my-cedar-experiment
```

End-to-end smoke test result with 9 variants × 10 plausible trials:

### Alternative: standalone (no etna-cli)

```sh
git clone https://github.com/alpaylan/cedar-etna.git
cd cedar-etna

# Witness replay (deterministic)
bash scripts/validate_lean_workload.sh

# Random search via marauders + the runner
python3 scripts/run_etna_experiment.py --strategy plausible --trials 20
```

`run_etna_experiment.py` mirrors what `etna experiment run` would do — apply
patches, build, run, restore — without needing etna-cli.

## Running

### Build the runner

```sh
cd workloads/Lean/Cedar/.cedar-spec/cedar-lean
lake build etna_cedar
```

This builds `EtnaCedar.Main` and its transitive Cedar imports — *not* the proof closure (`Cedar.Thm.*`). Some variants intentionally break those proofs when the patch reverses; only the implementation closure is required for ETNA.

### Run a witness (deterministic regression mode)

```sh
lake exe etna_cedar etna <PropertyName>
```

Emits exactly one JSON line on stdout:

```json
{"status":"passed","tests":1,"discards":0,"time":"0us","counterexample":null,"error":null,"tool":"etna","property":"DecimalParseNegativeSignPreserved"}
```

`status` ∈ `{passed, failed, aborted}`. Exit code is 0 except for argv-parse errors (exit 2) — etna2's `log_process_output` reads status from the JSON, never the exit code.

### Run random search (Plausible mode)

```sh
lake exe etna_cedar plausible <PropertyName>
```

Plausible drives `Gen` directly (200 trials, max-size 100) via `SampleableExt.interpSample`. Currently wired for the three properties whose inputs are primitive `String`:

- `DecimalParseNegativeSignPreserved`
- `DecimalParseNoUnderscore`
- `SmtEncodeStringBalancedQuotes` (IO)

Validator-soundness properties (`Schema`/`Request`/`Policies` inputs) report `status:"aborted"` until `Sampleable` instances are written for Cedar's structured types.

### Validate end-to-end

```sh
bash workloads/Lean/Cedar/scripts/validate_lean_workload.sh
```

Asserts the contract from `etna-ify/skills/validate/SKILL.md` (Lean variant):
1. `lake build etna_cedar` succeeds on base.
2. For every `[[tasks]]` in `etna.toml` whose `injection.kind = "patch"`:
   1. Reverse-apply the patch → buggy state.
   2. `lake build etna_cedar` succeeds.
   3. `lake exe etna_cedar etna <Property>` returns `status:"failed"`.
   4. Forward-apply patch → restore base; same runner returns `status:"passed"`.

Build target is `etna_cedar` (not full `Cedar`) so soundness fixes whose proofs in `Cedar/Thm/` are tightly coupled to the implementation can still be exercised at runtime.

## Adding a new variant

1. **Pick a candidate** from cedar-spec history. Triage by class — type-system / validator / SymCC bugs are higher-value than parser bugs (see `feedback_lean_workload_priorities` memory). Run `git -C .cedar-spec log --grep='fix\|sound\|incorrect' -- 'cedar-lean/Cedar/Validation/**' 'cedar-lean/Cedar/SymCC/**' 'cedar-lean/Cedar/TPE/**'` to surface candidates.
2. **Read the diff.** State the invariant the fix restores in one sentence. If you can't, drop the candidate.
3. **Verify the bug still reproduces.** Hand-edit the modern source to reintroduce the bug, run a quick `lake env lean --run` smoke test. If upstream Lean / batteries / dependencies have tightened in a way that masks the bug (cf. `String.toInt? "-"` returning `none` in Lean 4.29 vs `some 0` in 2023), record as `[[dropped]]` with reason `surface_changed`.
4. **Synthesize a patch.** `git format-patch -1 <fix-sha>` rarely applies cleanly to modern HEAD; instead, write the patch by hand against current source. Direction is buggy → fixed: `-` lines are the buggy code, `+` lines are the fix. Reverse-applying with `git apply -R` installs the bug.
5. **Write the property.** Pure, total, owned inputs, returns `PropertyResult`. The invariant should be sharp enough that the bug clearly violates it (cf. variant 5: `true \|\| ...` short-circuits the typechecker so only `checkEntities` differentiates).
6. **Write the witness** with frozen inputs that hit the mutated path. Run twice: base passes, reverse-applied patch fails. If both pass, the property is too coarse — sharpen it before moving on.
7. **Wire the runner.** Add a match arm in `EtnaCedar.Main.witnessFor`. For IO-typed witnesses (e.g. SymCC encoder bugs), use the `IO PropertyResult` form.
8. **Append to `etna.toml`.** Mirror an existing `[[tasks]]` block — the `mutations`, `[tasks.source]`, `[tasks.injection]`, `[tasks.bug]`, and `[[tasks.tasks]]` sections are all required.
9. **Commit scaffolding to `etna-base`.** Update `etna.toml`'s `base_commit` to the new SHA in `.cedar-spec`. Patches are independent of the base commit's content as long as they apply.
10. **Validate.** `bash scripts/validate_lean_workload.sh`. Regenerate docs with `etna workload doc`.

If a variant requires exposing a private symbol (cf. `Cedar.SymCC.Encoder.{encodeString, defineEntity, EncoderM}`), add the `public` modifier in the same `etna-base` commit. Cosmetic, no behavior change.

## Key gotchas

- **`Map.make` vs `Map.mk`.** Multi-entry `EntitySchema` literals must use `Map.make` to canonicalize; `EntitySchema.validateWellFormed` checks `Map.wellFormed`. Single-entry maps work either way.
- **Stub Solver for SymCC tests.** `defineEntity`'s enum branch never touches the solver streams, so `EtnaCedar.Main`'s `stubSolver` (writes routed to `/dev/null`) is sufficient. Properties that exercise the solver-output path (e.g. `defineRecord` for empty records) need a real string-buffer-backed `IO.FS.Stream`.
- **`lake build Cedar` vs `lake build etna_cedar`.** Soundness fixes' proofs in `Cedar/Thm/` are tightly coupled to the implementation. The validate script uses `etna_cedar` (impl-only closure) so variants whose buggy state breaks the proofs still validate at runtime.
- **Patches are synthetic.** `git format-patch -1 <fix-sha>` rarely applies due to Cedar's API drift (e.g. `String.split` → `String.splitToList`, `Environment` → `TypeEnv`). Hand-craft.

## Architecture notes

- **Runner artefacts on the base tree.** Untracked-file workflows break under `git stash` / branch switches, so `Cedar/Etna/`, `EtnaCedar/`, and the `lakefile.lean` `lean_exe etna_cedar` declaration are committed to the `etna-base` branch in `.cedar-spec`.
- **No checkpoint JSONs.** `etna.toml`, `Cedar/Etna/`, and `patches/` are the only durable state. `progress.jsonl` is per-run scratch.
- **JSON-on-stdout contract.** `EtnaCedar.Main` always exits 0 except on argv parse errors (exit 2). etna2's `log_process_output` parses each stdout line as JSON and merges it into the experiment context — non-zero exit becomes `status:aborted` regardless of what was printed.
